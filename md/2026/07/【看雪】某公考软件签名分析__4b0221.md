> 💡 **AI 总结（key-points）**
>
> 公考软件的sign参数生成机制是将请求参数归一化后，拼接硬编码secret字符串并进行MD5哈希。
> 
> - **签名生成流程**：所有请求参数按key字典序升序排列，拼接成`k=v&...&`格式，经`URLEncoder.encode(URLDecoder.decode(...))`归一化后，在native层追加硬编码secret字符串。
> - **native实现细节**：`libsign.so`中的`getSignByJNI`函数根据服务端type（1、2、3）选择不同secret（如type=1对应`GO1GgDeC1qIr7KcuGfZIGTHk5R3RM1KT`），对拼接串进行MD5哈希输出32位小写十六进制。
> - **防篡改机制**：native层通过`checkSignature`校验包名必须为`com.maidu.gkld`，且APK签名证书的hashCode等于`-1281511612`，否则返回"sign error"。
> - **抓包前准备**：应用检测代理/VPN（`XUtil.isProxyEnabled`），需使用Frida hook该函数返回false以绕过检测，否则会伪造假响应。

本文仅用于安全研究与学习交流，分析对象为公开发行的 Android 应用。请勿将文中技术用于任何非法用途，由此产生的一切后果与作者无关。

## 目标

本次分析对象为 公考\*\*6.5.3.0（包名 `com.****.gkld` ），目标是其接口请求里的 `sign` 参数

全程用到的工具：

## 抓包

先挂代理抓包，拿到登录验证码校验接口的请求：

```python
POST https://api.****.com/api/***/login/captcha/verify
user-agent: okhttp/4.5.0
x-version: 6.5.3.0
x-os: 29
x-device: android

phone=***************&captcha=***&userId=0&uuid=***************
&oaid=***&timestamp=1784614939&sign=abff36fff6022292cc34a4f0ee5010a2
&from_device=android&appid=8tQhhq32HIxOH72p&appLaunchChannel=default&devVersion=6.5.3.0&osVersion=29
```

关注点很明确： `sign` 是个 32 位小写十六进制串，长度特征像 MD5。下面要定位它的生成位置。

> 顺带一提：这个 app 对代理/VPN 有检测（ `com.midas.base.utils.XUtil.isProxyEnabled` ，命中会让拦截器伪造 `检测到代理，请关闭代理` 的假响应），抓包前需要先用 Frida 把它绕过。这不是本文重点，后面附一节简单带过。

## 定位签名生成位置

`sign` 是所有请求都带的公共参数，通常在 OkHttp 拦截器里统一加。用 jadx 搜 `sign` 方法名，很快定位到网络层的 `com.midas.net.interceptor.ParamInterceptor` 。它的 `intercept()` 里对 POST 请求做了统一处理：

```java
// ParamInterceptor.intercept() 关键片段
String strValueOf = String.valueOf(System.currentTimeMillis() / 1000);   // 秒级时间戳
...
// 拼接参与签名的参数串
String str = (((strRemoveParam2
        + "&appid=" + ServerFactory.getInstance().getServer().getAppId())
        + "&devVersion=" + NetWorkUtils.getNetVersion())
        + "&osVersion=" + Build.VERSION.SDK_INT)
        + "&appLaunchChannel=" + ch;
String sign = getSign(str, strValueOf);          // ← 生成 sign
...
builder.addEncoded("timestamp", strValueOf)
       .addEncoded(f.Y, sign)                     // f.Y == "sign"
       .addEncoded("from_device", "android")
       ...
```

其中 `strRemoveParam2` 是原始请求体去掉 `userId/uuid/oaid` 后，再补上 `from_device=android` 、 `token` 、 `userId` 、 `uuid` 、 `oaid` 得到的串。

继续跟 `getSign` ：

```java
public String getSign(String parmes, String currentTimeMillis) throws IOException {
    HashMap map = new HashMap();
    for (String kv : parmes.split("&")) {           // 拆成 map
        String[] p = kv.split("=");
        map.put(p[0], p.length == 1 ? "" : p[1]);
    }
    map.put("timestamp", currentTimeMillis);         // 补上 timestamp
    return SecretUtils.getSign(map);                 // ← 交给 SecretUtils
}
```

## 参数归一化：SecretUtils.getSign

```java
public static String getSign(HashMap<String, String> map) {
    StringBuilder sb = new StringBuilder();
    ArrayList arrayList = new ArrayList(map.keySet());
    Collections.sort(arrayList);                     // key 字典序排序
    for (int i = 0; i < arrayList.size(); i++) {
        sb.append(arrayList.get(i) + "=" + map.get(arrayList.get(i)) + "&");   // k=v&...&
    }
    return SignUtils.getSignByJNI(
            URLEncoder.encode(URLDecoder.decode(sb.toString())),   // URL 归一化
            CommonContext.get(),
            ServerFactory.getInstance().getServer().getType());    // 服务器 type
}
```

到这里 Java 层的处理很清晰了：

1.  所有参数（含 `timestamp` ）按 **key 字典序升序** 排列；
2.  拼成 `k=v&k=v&...&` （ **末尾带 `&`** ）；
3.  整串做 `URLEncoder.encode(URLDecoder.decode(...))` 归一化（先解码再统一编码， `=` → `%3D` 、 `&` → `%26` ）；
4.  把归一化后的串、 `Context` 、 `type` 三个参数丢给 **native 函数 `SignUtils.getSignByJNI`** 。

真正的算法在 native。

## 确定 native 库

看 `com.maidu.gkld.jni.SignUtils` ：

```java
public class SignUtils {
    static {
        System.loadLibrary(f.Y);      // f.Y == "sign"  ->  libsign.so
    }
    public static native String getSignByJNI(String params, Context context, int type);
}
```

`f.Y` 这个常量同时被用作 `sign` 字段名和库名，结合 APK 里存在 `lib/arm64-v8a/libsign.so` ，确定：

-   native 库： **libsign.so**
-   native 函数： `Java_com_maidu_gkld_jni_SignUtils_getSignByJNI`

## IDA 分析 libsign.so

用 IDA 打开 `libsign.so` ，直接在导出表找到静态注册的 JNI 函数 `Java_com_maidu_gkld_jni_SignUtils_getSignByJNI` （ `0x3efe0` ，无 `JNI_OnLoad` 动态注册）。反编译如下：

![image-20260724103712380](https://cdn.jsdelivr.net/gh/zhiyu-zeng/img@main/img/2026/07/3fab3316499c6850.png)  
算法已经很明显了

```c
jstring getSignByJNI(JNIEnv *env, jclass clazz, jstring params, jobject context, int type)
{
    __android_log_print(4, "KWAN_JNI", "Java_com_maidu_gkld_jni_SignUtils_getSignByJNI");

    // (1) 防篡改校验: 校验包名 + 签名证书 hashCode
    if ((unsigned)checkSignature(env, context) >= 0xFFFFFFFE) {
        __android_log_print(4, "KWAN_JNI", "hashcode error");
        return env->NewStringUTF("sign error");
    }

    // (2) 取 String.concat 方法, 按 type 选 secret
    jmethodID concat = env->GetMethodID(String, "concat", "(Ljava/lang/String;)Ljava/lang/String;");
    jstring secret = env->NewStringUTF("secret%3DGO1GgDeC1qIr7KcuGfZIGTHk5R3RM1KT");  // 默认
    if (type == 3) secret = env->NewStringUTF("secret%3DrCkPSJRt8Il5OMxuW8EWBlhLVBvXOXZI");
    if (type == 2) secret = env->NewStringUTF("secret%3DcHCQHgYoHN5VBwzmlcyjxxgfVAExdyX7");
    if (type == 1) secret = env->NewStringUTF("secret%3DGO1GgDeC1qIr7KcuGfZIGTHk5R3RM1KT");

    // (3) params.concat(secret)  —— 用 JNI 调 Java 的 String.concat 拼接
    jstring joined = env->CallObjectMethod(params, concat, secret);
    const char *s = env->GetStringUTFChars(joined, 0);

    // (4) 对拼接后的串做 MD5
    std::string str(s);
    MD5 md5(str);
    std::string hex = md5.hexdigest();

    return env->NewStringUTF(hex.c_str());
}
```

### 三个硬编码 secret

`%3D` 就是 `=` ，拼接时保持编码形态。按服务端 `type` 选择：

| type | 追加字符串 | 实际 secret |
| --- | --- | --- |
| 1   | `secret%3DGO1GgDeC1qIr7KcuGfZIGTHk5R3RM1KT` | `GO1GgDeC1qIr7KcuGfZIGTHk5R3RM1KT` |
| 2   | `secret%3DcHCQHgYoHN5VBwzmlcyjxxgfVAExdyX7` | `cHCQHgYoHN5VBwzmlcyjxxgfVAExdyX7` |
| 3   | `secret%3DrCkPSJRt8Il5OMxuW8EWBlhLVBvXOXZI` | `rCkPSJRt8Il5OMxuW8EWBlhLVBvXOXZI` |

另外说一句，这个 so 的 `getSignByJNI` **没有做控制流平坦化（ollvm）** ，反编译出来只有 20 多个基本块、结构清晰，分析成本比某些 ollvm 加固的 so 低很多。

## 防篡改 checkSignature

进 sign 计算前会先过 `checkSignature` （ `0x3d1b8` ），整理后逻辑：

```c
int checkSignature(JNIEnv *env, jobject context)
{
    // 通过 context 反射拿到 PackageInfo.signatures[0]
    ... getPackageManager -> getPackageInfo(pkg, GET_SIGNATURES=64) -> signatures[0] ...
    int hash = signatures[0].hashCode();
    const char *pkg = getPackageName();

    if (strcmp(pkg, "com.maidu.gkld") != 0) return 0xFFFFFFFF;   // 包名不对 -> 失败
    if (hash == -1281511612)  return -1281511612;                // 证书正确 -> 通过
    else                      return -2;                          // 证书不对 -> 失败
}
```

调用方判断 `checkSignature(...) >= 0xFFFFFFFE` 即为失败，返回 `"sign error"` 。也就是要求：

-   包名必须是 `com.maidu.gkld`
-   APK 签名证书 `signatures[0].hashCode()` 必须等于 `-1281511612` （ `0xB3A6A6C4` ）

这是防重打包/重签名的措施。

## 算法还原

综合 Java 层 + native，signature 的完整算法为：

```python
sign = MD5_hex( URLEncoder.encode(URLDecoder.decode( 排序拼接串 )) + "secret%3D<SECRET>" )
```

其中：

-   **排序拼接串** = 所有参数（含 `timestamp` ）按 key 字典序升序，拼成 `k=v&k=v&...&` （末尾带 `&` ）
-   整串经 Java `URLEncoder.encode(URLDecoder.decode(...))` 归一化
-   native 里再 `.concat("secret%3D<SECRET>")`
-   对最终串做标准 **MD5** ，输出 32 位小写十六进制

## Frida 校验

如果想在真机上直接看 native 的输入输出（也能确认 `type` ），hook `getSignByJNI` 即可：

```javascript
Java.perform(function () {
    var S = Java.use("com.maidu.gkld.jni.SignUtils");
    S.getSignByJNI.implementation = function (params, ctx, type) {
        var r = this.getSignByJNI(params, ctx, type);
        console.log("[sign] type=" + type + "\n  params=" + params + "\n  sign=" + r);
        return r;
    };
});
```

实测抓到的 `params` 形如 `appLaunchChannel%3Ddefault%26appid%3D...%26timestamp%3D...%26` ， `type=1`

### 附：代理/VPN 检测绕过

抓包前需绕过 `com.midas.base.utils.XUtil.isProxyEnabled` （检查 `http(s).proxyHost` 、 `ProxySelector` 、VPN 网卡 `tun0/ppp0` 、 `NetworkCapabilities.TRANSPORT_VPN` ），命中会让拦截器伪造假响应。Frida 直接让它恒返回 false 即可：

```javascript
Java.perform(function () {
    var XUtil = Java.use("com.midas.base.utils.XUtil");
    XUtil.isProxyEnabled.overloads.forEach(function (ov) {
        ov.implementation = function () { return false; };
    });
});
```
